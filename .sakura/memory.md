# 项目记忆

累计反思 10 次

## 核心经验教训

### 1. 异步资源管理（关键）
- **"释放后立即清理"陷阱**：`asyncio.Lock` 的 `release()` 只是唤醒等待者，不是执行等待者。禁止在同步代码中基于 `locked()` 返回值做"状态判断"并立即清理字典条目。
- **异步上下文管理器优先**：对于需要严格配对获取/释放的异步资源（如锁），应强制使用 `async with`，消除手动管理错误。
- **资源生命周期策略**：对"字典/列表 append"操作，必须思考其逆操作或清理机制，防止只增不减导致内存泄漏。

### 2. 配置管理规范
- **配置类型安全强制规范**：所有从配置读取的数值型参数，必须通过统一辅助方法（`_cfg_int`, `_cfg_float`, `_cfg_bool`）进行读取，禁止裸 `int()/float()` 转换。
- **配置范围校验**：数值型配置应明确有效范围并在校验时进行边界检查。
- **分离持久化配置与运行时缓存**：使用独立实例变量（如 `self._cache_*`）存储运行时数据，避免污染持久化配置。

### 3. 增量审查方法论
- **保持全局观**：增量审查必须将变更置于整个文件、整个请求生命周期中理解，检查新代码与旧代码在更大架构下的潜在冲突。
- **深度优于广度**：有时一个方法的深入审查足以揭示系统性问题。
- **模式一致性检查**：发现一处"坏味道"被修复后，必须主动在同一模块中查找相同模式，确保问题彻底解决。

### 4. 防御性编程
- **异常处理路径中的变量定义**：涉及 `try/except` 块且异常分支使用 `try` 内变量时，必须审查变量是否可能在 `try` 块赋值前被异常中断。
- **配置值的"恶意输入"压力测试**：假设配置值可能为空、为负、为极大值、为特殊字符，审查正则 ReDoS、延迟负值、Token阈值空列表等问题。
- **防御性编程的边界**：错误处理代码本身必须比主流程更健壮，避免"雪上加霜"。

### 5. 日志与错误恢复
- **日志级别哲学**：对因外部输入导致的非致命但需关注的失败，`warning` 是合理的；调试信息用 `debug`；需要关注的故障用 `warning/error`。
- **错误日志上下文**：关键错误日志应包含足以定位问题的核心业务变量（如 session_key、provider_type）。
- **降级策略**：审查错误恢复机制，确保系统在异常时有合理的降级而非静默失败。

### 6. 架构观察
- **配置中心化但分散读取**：`main.py` 集中定义配置但分散读取，容易导致配置处理逻辑碎片化，需统一配置辅助方法。
- **模块职责清晰但交互紧密**：`manager.py` 负责状态，`main.py` 负责流程编排，但基础服务散落各处，降低内聚性。
- **上下文管理核心地位**：`ChatContextManager` 承担锁管理、Token缓存、消息记录等多重职责，未来可能需要拆分为更专注的组件。

### 7. 审查规则建议
- **异步锁使用规范**：禁止使用 `asyncio.get_event_loop()` 进行状态检查，统一使用 `asyncio.get_running_loop()`。
- **重复定义检测**：审查中强制检查方法/函数是否存在重复定义，尤其关注复制粘贴引入的遗漏。
- **前端属性输出安全**：`escapeHtml` 应用于所有动态生成的 HTML 属性值，不仅是文本内容。

## 项目特定规范
- **AstrBot插件接口**：`record_passive_message` 等方法行为变更需确保与调用方预期一致。
- **配置管理**：布尔类型配置项应严格接受 `0/1`、`true/false`，其他值视为无效。
- **插件兼容性文档**：当代码兼容多种外部接口时，必须在文档字符串中明确说明兼容的历史原因和演进方向。
