## 代码审查反思：PR #3，提交 4ddd6b3

### 1. 审查质量评估

*   **覆盖度**：
    *   **基本覆盖，但存在重大遗漏**。审查准确指出了新变更的积极方面（防抖、清理）和多个遗留问题。**然而，遗漏了一个与项目核心规范直接冲突的关键问题**：`proactive/manager.py` 中新增的 `_flush_registry()` 方法使用了**同步文件 I/O** (`self._registry.write_text(...)`)。这直接违反了项目记忆中“异步I/O规范”中“在`async`上下文中禁止使用同步文件I/O”的强制要求，且在 `close()` 这个同步方法中调用异步上下文管理器的 `flush` 也是可疑的。这是一个`major`级别的遗漏。
    *   对于安全问题（`httponly=False`, `CORS *`）和健壮性问题（`JSONDecodeError`）的标记是充分的。

*   **准确度**：
    *   **大部分问题标记准确，但严重程度有争议**。将 `task_id` 构造问题标记为 `major` 可能过于严格。它确实会导致任务覆盖，但在当前场景下（相同会话+相同延迟）覆盖是否构成“主要”风险，取决于业务逻辑。标记为 `medium` 或 `high` 可能更合适。配置函数重复 (`DRY`) 标记为 `major` 是合理的，因为它影响可维护性和一致性。

*   **完整性**：
    *   审查**很好地理解了本次变更的意图**（优化 I/O 和内存），并正确关联了历史问题。但它**未能将变更置于整个模块的异步上下文约束中理解**，导致遗漏了同步 I/O 这个根本性问题。审查摘要中提到的“轮数触发时仍直接调用 `_save_registry` 破坏防抖机制”是正确的，但审查没有进一步追问：即便改为 `_mark_dirty()`，其底层的 `_flush_registry` 实现本身是否合规？

### 2. 发现的模式

*   **代码模式**：
    *   **同步/异步混用**：项目（特别是 `manager.py` 模块）存在明显的同步文件操作习惯（如 `_save_registry`），即使在异步框架中也未完全改造。本次提交虽引入了防抖，但核心写入方法 `write_text` 仍是同步的，这是深层技术债务。
    *   **配置处理碎片化**：`_cfg_int`, `_cfg_bool` 等辅助函数在多个 `manager.py` 中重复定义，表明缺乏一个集中的、被依赖的基础工具模块。这是典型的“实用主义”代码演化结果。
    *   **防御性编程不均**：对 Web 层的输入验证（`JSONDecodeError`）关注较高，但对内部状态操作（如注册表构造）的健壮性关注不足（如 `task_id` 唯一性）。

*   **架构观察**：
    *   **“Manager”模式职责过重**：`proactive/manager.py` 同时负责任务调度、注册表状态管理、持久化（I/O）、内存清理。这导致其同时面临状态一致性、I/O 性能和代码复杂度的挑战。
    *   **基础设施与业务逻辑脱节**：像“安全的 JSON 解析”、“配置类型转换”、“安全的文件写入”这类横切关注点，没有形成统一的基础设施层，导致各处重复解决或遗漏。

### 3. 规范完善建议

*   **新增/修改审查规则**：
    1.  **强制异步 I/O 审查**：在审查任何 `manager.py` 或 `service.py` 等常驻服务模块时，必须将文件读写操作列为检查项，强制使用 `aiofiles` 或 `run_in_executor`。此条应置顶。
    2.  **唯一性 ID 生成规范**：对于可能用于去重、索引或作为缓存键的 ID，审查其生成策略是否保证全局唯一（如 UUID, 雪花算法等），避免因碰撞导致逻辑错误。
    3.  **配置辅助方法集中化**：将配置读取、类型转换、默认值处理的通用辅助方法提取到 `utils/config.py`，作为项目规范。审查时检查新模块是否遵守此规范。

*   **项目特有规范要点**：
    *   **重申异步上下文约束**：在项目核心规范中明确，“所有 Manager 的 `save/update/flush` 等涉及持久化的公开方法，若为 `async`，其内部必须使用异步 I/O；若为同步，则必须通过线程池执行器包装文件操作。”
    *   **WebUI 安全基线细化**：明确指出在当前项目假设（可能内网部署）下，`httponly=False` 和 `CORS *` 是否为**已知且可接受的风险**。如果是，应在文档中显式声明；如果不是，应列为 `blocker` 级别问题。

### 4. 经验教训

*   **技术债的传导**：本次审查发现，为解决一个技术债（I/O 频繁）引入的方案（防抖机制），其自身实现却违反了另一条更基础的技术债规范（异步 I/O）。**教训**：审查解决旧问题的方案时，必须将其与项目现存的所有相关规范进行交叉验证，防止“新债抵旧债”。
*   **“死代码”审查**：`_registry_save_interval` 未使用的问题，可能是重构或优化时的遗留。审查时，发现此类“定义未使用”的代码，除了删除建议，还应追问：“它是否暗示了一个未完成或半途而废的优化计划？”
*   **增量审查的陷阱**：本次审查很好地完成了对“增量”的评估，但**未能将增量置于整个文件的异步上下文中进行最终校验**。教训：增量审查完成后，应花费片刻，以“全局视角”快速扫视变更所在类/模块，确认新旧代码在架构模式上的一致性。

### 5. 增量审查反思

*   **解决历史问题**：**是**。提交明确且有效地解决了“频繁写盘”和“token 内存增长”两个历史 `major` 问题，方案（脏标记、定期清理）合理。
*   **引入新风险**：
    1.  **引入了明确的规范违反**：`_flush_registry()` 中的同步文件 I/O。
    2.  **引入了潜在的数据一致性风险**：在 `on_turn` 等高频调用中设置 `_registry_dirty`，但在非高频的定时 flush 中写入，如果进程在两次 flush 间崩溃，可能丢失部分“脏”状态。
*   **审查覆盖度变化**：**有下降趋势**。因为聚焦于 `feat(proactive)` 这一明确的提交标题，审查重点容易落在 `proactive/manager.py` 的变更逻辑上。而 `web/server.py` 的变更（+22/-2）相对较小，可能只是扫了一眼。对于 `server.py` 中**未变更但已存在的历史问题**（如 `JSONDecodeError`），审查的“发现”更多是基于记忆的“提醒”，而非本次增量审查的“新发现”。这可能导致对 `server.py` 部分的审查深度不足。需要意识到，增量审查的对象不仅是“变更的代码”，也包括“变更对未变更代码的影响”。

**总结**：本次审查在功能逻辑和解决历史问题方面表现良好，但因未能严格应用“异步I/O”这一关键约束，导致遗漏了一个架构级的合规性问题。未来的审查需加强**跨文件、跨规范**的联动检查，并对“增量”与“存量”问题进行更清晰的权重划分。
